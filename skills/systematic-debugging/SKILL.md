# 系统化调试（Systematic Debugging）

借鉴 superpowers 的 systematic-debugging：**系统化优于碰运气，证据优于猜测**。

## 流程

### 1. 复现
先稳定复现问题。记录：触发步骤、输入、预期 vs 实际、报错栈/日志。无法复现就先收集更多观测点（加日志、读 console / 网络 / 日志文件）。

### 2. 假设
基于证据列出 1–3 个**具体、可证伪**的假设（"X 为空导致 NPE" 而非 "可能哪里有问题"）。按可能性排序。

### 3. 验证
**逐个验证假设，一次只改一个变量**。用最小手段确认：加日志、断点、缩小输入、二分注释。在确证根因前**不要动业务代码**。

### 4. 定位根因
找到真正的根因（root cause），而非表象。问"为什么会这样"直到触底——常常表层错误下面还有一层。

### 5. 修复
针对根因做最小修复。考虑：是否还有同类隐患？是否需要防御性校验？

### 6. 复验
重跑原始复现步骤确认已修复；确认没有引入回归；`mvn clean compile` 通过。若处于 SDD 流程，对照相关 spec 场景复查。

## 禁忌

- ❌ 不复现就改码、改完"应该好了吧"。
- ❌ 一次改一堆东西，分不清是哪个起了作用。
- ❌ 只压住报错（catch 吞异常）而不查根因。
- ❌ 用"重启/清缓存就好了"代替理解。

## 调试可用观测手段（本项目）

- 读日志：工作区 `logs/javaclaw.log` / `logs/task.log`
- 浏览器场景：read_console_messages（按 pattern 过滤）/ read_network_requests
- 加临时日志后用 `mvn clean compile` 重跑
