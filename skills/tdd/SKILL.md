# 测试驱动开发（TDD）

借鉴 superpowers 的 test-driven-development。每个 spec 场景对应至少一个测试。

## RED-GREEN-REFACTOR 循环

1. **RED（红）**：先写一个**会失败**的测试，描述期望行为（对应一个 spec 场景）。运行它，确认它确实因为功能缺失而失败——而不是因为编译错或写错断言。
2. **GREEN（绿）**：写**刚好够用**的实现让测试通过。不提前优化、不加多余功能。
3. **REFACTOR（重构）**：测试保持绿的前提下清理代码——去重、命名、抽取。每次重构后重跑测试确保仍绿。

循环往复，直到该 spec 的所有场景都有绿测试覆盖。

## 准则

- **先红再绿**：没看到测试失败过，就不知道它真的在测东西。
- **一次一个行为**：每个测试聚焦单一可观测行为（对照 Given/When/Then）。
- **测试即规格的可执行版**：测试名与断言应能映射回 spec 场景。
- **不为通过而改测试**：实现不达标时改实现，而不是放宽测试。

## 与本项目的现实约束

本项目（JavaClaw）**尚未集成测试框架**（见 CLAUDE.md）。因此：
- 若目标模块已有/可引入测试手段，按标准 TDD 走。
- 若暂无测试框架，退而求其次：为关键逻辑写一个**可独立运行的最小验证程序/main 方法或临时断言**先证伪、再实现、再清理；并用 `mvn clean compile` 保证编译。把这一过程记录到 tasks.md 的测试小节。
- 不要臆造 `mvn test` / `mvn verify`（项目未配置）。
