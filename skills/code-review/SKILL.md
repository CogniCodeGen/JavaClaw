# 代码评审（Code Review）

借鉴 superpowers 的 requesting-code-review：实现完成不等于完成，**先过质量门禁**。严重问题阻断交付。

## 评审两条主线

### 一、计划符合度（对照规格）
- 是否实现了 `proposal.md` 声明的全部变更点？有没有遗漏？
- 行为是否满足 `specs/{能力}/spec.md` 的每个场景（Given/When/Then）？
- `tasks.md` 是否全部勾选完成？未完成项要么做完、要么明确说明为何延后。
- 有没有**夹带范围外改动**？有则回到提案阶段补登记或剔除。

### 二、代码质量
- **正确性**：边界条件、空值、异常路径、并发/线程（本项目：UI 更新须 `Platform.runLater`，Agent 调用在 `boundedElastic`）。
- **简洁与复用**：有无重复逻辑可抽取？有无更简单写法？是否复用了既有工具/端口而非另起炉灶？
- **一致性**：命名、注释密度、风格是否贴合周边代码？中文注释/提示/日志（项目约定）。
- **资源与清理**：流/连接/订阅是否正确释放（参考 `AtomicDisposable`）？
- **错误处理**：`@Tool` 方法是否返回 `ToolResponse.success/error/timeout` 规范格式？

## 严重度分级

- **阻断（必须改）**：功能不符 spec、会崩溃/数据错误、范围越界。
- **建议（宜改）**：可读性、重复、轻微低效。
- **可选（可记 TODO）**：风格偏好、未来优化。

阻断项未清零 **不得**标记任务完成。

## 产出

给出结构化评审结论：按严重度列出问题 + 文件:行号 + 修复建议。逐条修复阻断项后复验（`mvn clean compile` + spec 场景）。通过后回到「规格驱动开发」做归档。
