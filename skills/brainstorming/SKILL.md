# 需求澄清（Brainstorming）

借鉴 superpowers 的 brainstorming：**先理解，再设计，最后才编码**。澄清不充分就开工是返工的头号原因。

## 准则

1. **不要立刻写代码**。先确认自己真正理解了用户想要什么。
2. **一次问少量、高价值的问题**（2–4 个），不要一口气抛长问卷。
3. **分段呈现设计**：把构想拆成小块逐段给用户确认，而非一次甩出完整方案。
4. **暴露假设**：把你的隐含假设显式说出来让用户校正。

## 要澄清的维度（按需取用）

- **目标**：成功长什么样？最终交付物是什么？
- **边界**：范围内 / 范围外分别是什么？哪些明确不做？
- **验收标准**：怎样算"完成"？可观测、可验证的判据是什么？
- **约束**：技术栈、性能、兼容性、依赖、时限。
- **取向选择**：当有多条实现路径时，列出选项 + 各自取舍，让用户拍板。
- **优先级**：若需求多，哪些必须先有（MVP），哪些可后置。

## 产出

澄清收敛后，用一段话回述确认结果：目标 + 边界 + 验收标准 + 关键决策。这段回述将作为 `openspec/changes/{变更}/proposal.md` 的素材。

## 下一步
