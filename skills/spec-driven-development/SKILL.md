# 规格驱动开发（Spec-Driven Development）

借鉴 OpenSpec 与 superpowers 方法论：**规格是唯一真相，先对齐再编码**。任何超出"一行修改"的编程任务都应先产出规格产物，再实现。

## 何时启用

- 新增功能、能力或模块
- 重大重构 / 架构调整 / 接口变更
- 跨多个文件、需要权衡设计方案的修改
- 复杂或反复出现的 bug（配合「系统化调试」）

> 极简的一次性查询/单行修改可跳过本流程，直接实现。能省则省，但**新增能力一律先立规格**。

## 产物目录约定

所有规格产物落在工作区根目录的 `.agent/openspec/` 下（与技能同处 `.agent/`，集中管理）：

```
.agent/openspec/
├── project.md                       # 项目级长期约定（技术栈/规范/术语），可选、跨变更复用
├── changes/
│   └── {YYYYMMDD-变更短名}/          # 一次变更一个目录，短名用 kebab-case
│       ├── proposal.md              # 必须：为什么做 + 改什么（高层）
│       ├── design.md                # 仅复杂变更需要：技术方案、权衡、取舍
│       ├── tasks.md                 # 必须：实现清单（可勾选的有序步骤）
│       └── specs/
│           └── {能力名}/spec.md     # 必须：需求 + 场景（Given/When/Then）
└── specs/                           # 已归档的"当前真相"：变更完成后从 changes/ 合并而来
    └── {能力名}/spec.md
```

产物模板见本技能 references/ 下的 `*-template.md`，创建文件时严格套用其结构。用文件写入工具（系统文件工具）创建与更新这些 Markdown，**不要只在对话里口头描述**。

## 工作流程（六阶段）

### 1. 澄清（Clarify）
调用 `skill_read("需求澄清")` 并据其指令向用户提问，直到目标、边界、验收标准明确。**澄清未完成不进入下一步。**

### 2. 提案（Propose）
在 `.agent/openspec/changes/{YYYYMMDD-短名}/proposal.md` 写下：
- **为什么**：动机、要解决的问题、不做会怎样
- **改什么**：高层变更点（新增/修改/删除哪些能力）
- **不改什么**：明确划出范围外的内容，防止范围蔓延

向用户展示提案要点并征求确认后再继续。

### 3. 规格（Specify）
为每个受影响能力在 `.agent/openspec/changes/{变更}/specs/{能力名}/spec.md` 写需求与场景：
- 用 `## 需求：xxx` 列出可验证的需求条目
- 每条需求下用 `### 场景：xxx` + Given/When/Then 描述可观测行为
- 规格描述"做什么"和"对外行为"，不写实现细节

### 4. 设计（Design，按需）
仅当方案有非平凡技术权衡（选型、并发、数据结构、向后兼容）时，写 `design.md`：列出候选方案、各自取舍、最终选择与理由。简单变更跳过此步并在 proposal 里注明"无需设计文档"。

### 5. 任务清单（Tasks）
在 `tasks.md` 把实现拆成有序、细粒度（每项 2–5 分钟可完成）、可勾选的步骤，每项写明：动作、涉及文件路径、完成判据。形如：
```
- [ ] 1. 在 Xxx 类新增 yyy 方法（src/.../Xxx.java）— 判据：方法签名与 spec 场景一致
```

### 6. 实现与验证（Implement & Verify）
逐项推进 tasks.md，**每完成一项把 `[ ]` 改成 `[x]`**。实现时：
- 涉及新逻辑 → 先 `skill_read("测试驱动开发")` 走 RED-GREEN-REFACTOR
- 遇到 bug/异常 → `skill_read("系统化调试")`，先假设后验证，禁止臆测改码
- 编码完成 → `skill_read("代码评审")` 对照 proposal/spec/tasks 自查质量门禁
- 改动后按项目约定验证（本项目用 `mvn clean compile` 验编译，不要臆造 `mvn test`）

## 完成后归档

变更通过验收后，把 `.agent/openspec/changes/{变更}/specs/` 下的规格合并进顶层 `.agent/openspec/specs/`（更新"当前真相"），并在变更目录的 `proposal.md` 末尾标注「已完成 + 日期」。保留变更目录作为历史审计。

> 同时按「架构同步」要求更新 `.agent/ARCHITECTURE.md`：把本次代码变更后的架构逻辑、关键产出同步进去，并在其「变更日志」追加一条。

## 铁律

- **证据优于断言**：声称"已修复/已实现"必须有验证依据（编译通过、行为符合 spec 场景）。
- **范围即契约**：超出 proposal 范围的改动要回到第 2 步更新提案，不得夹带。
- **产物先行**：proposal / spec / tasks 缺一不可（design 按需）；它们落盘后才动代码。
- **中文优先**：所有产物、注释、提示均用中文（与本项目约定一致）。
